As vector search workloads shift from experimental prototypes to production-critical infrastructure, the performance characteristics of the underlying database engine have direct financial consequences. A system that takes 45 minutes to rebuild an index after a bulk insert is not just a technical annoyance; it locks up engineering time, delays feature rollouts, and forces teams to over-provision compute to hit latency SLOs during the rebuild window. In August 2024, Zilliz published a series of benchmarks comparing Milvus 2.4’s GPU-accelerated index construction against a widely-used Elasticsearch vector deployment. The numbers surfaced a gap that changes the calculus for teams managing collections above the 10-million-vector mark. With GPU instances on AWS priced at US$3.06 per hour for an on-demand g5.2xlarge (NVIDIA A10G, 24 GB VRAM) as of September 2024, the decision to use GPU-accelerated indexing is no longer confined to hyperscaler budgets. Independent developers and mid-stage startups running retrieval-augmented generation (RAG) pipelines, semantic search, or multi-modal retrieval now have a measurable cost-performance trade-off to evaluate. The Milvus 2.4 release, dated June 2024, introduced a rewritten brute-force index builder that offloads computation to CUDA cores, and the benchmark results from August 15, 2024, provide the first concrete evidence that GPU index construction can reduce wall-clock time by an order of magnitude on commodity cloud hardware. This article examines those benchmarks, compares them against CPU-bound Elasticsearch vector indexing, and provides a framework for determining when the GPU tax is worth paying.

## Index Build Throughput: Raw Numbers from the August 2024 Benchmarks

The core benchmark scenario involved building an IVF_FLAT index on the Deep-1B dataset, a standard large-scale ANN benchmark collection containing 1 billion 96-dimensional vectors. The test measured end-to-end wall-clock time from raw vectors to a fully queryable index on a single node. Milvus 2.4 was configured with the GPU index builder enabled, running on an AWS g5.2xlarge instance (8 vCPUs, 32 GB RAM, 1× NVIDIA A10G). The Elasticsearch comparison point used version 8.14.1 with the dense_vector field type and HNSW indexing, running on an equivalent CPU-only instance with 8 vCPUs and 32 GB RAM. The dataset size for the head-to-head was capped at 50 million vectors to keep the Elasticsearch indexing time within a practical bound.

### Milvus 2.4 GPU Index Build: 50 Million Vectors in 8.2 Minutes

On the 50-million-vector subset, Milvus 2.4 with GPU acceleration completed IVF_FLAT index construction in 8.2 minutes. The GPU builder saturated the A10G’s compute capacity, hitting approximately 85% CUDA core utilization during the brute-force clustering phase. Memory consumption on the GPU side peaked at 18.4 GB of the available 24 GB VRAM, leaving headroom for larger batches without spilling to system RAM. The index file written to disk was 9.6 GB. Post-build query latency at 99th percentile (p99) for a top-10 ANN search with nprobe=32 measured 4.7 milliseconds on the same hardware.

### Elasticsearch 8.14.1 HNSW Index: 50 Million Vectors in 97 Minutes

Elasticsearch 8.14.1, using its native HNSW implementation with default parameters (m=16, ef_construction=100), required 97 minutes to build an index on the identical 50-million-vector dataset. The process was CPU-bound on all 8 vCPUs, with no GPU offload path available. Memory consumption during index construction fluctuated between 26 GB and 31 GB, occasionally triggering JVM garbage collection pauses that added approximately 4 minutes of cumulative stall time across the build. The resulting index size on disk was 14.2 GB. Query latency at p99 for the equivalent top-10 search measured 8.1 milliseconds on the same instance type, though this comparison is confounded by the different index algorithms (IVF_FLAT vs HNSW) and is included here only to contextualize the end-to-end operational profile, not as a direct algorithmic latency benchmark.

### Scaling to 100 Million Vectors: Where Elasticsearch Timed Out

The benchmark protocol set a 6-hour timeout threshold for index builds, reflecting a practical overnight batch window. Milvus 2.4 GPU indexed 100 million vectors in 19.3 minutes, scaling sub-linearly relative to the 50-million-vector run. Elasticsearch 8.14.1 exceeded the 6-hour timeout on the 100-million-vector dataset and did not complete. The test was terminated at 387 minutes with index build progress estimated at 62% based on segment merge logs. This result is consistent with HNSW’s known super-linear build complexity; each additional vector increases the number of distance computations in the graph construction phase, and without GPU acceleration, the CPU-only path becomes untenable for large collections within bounded time windows.

## The Cost Equation: GPU Instance Pricing vs Engineering Time

The raw speed numbers are compelling, but a build-time advantage only translates to a buying decision when mapped to actual cloud spend and operational constraints. As of September 2024, AWS on-demand pricing for the instance types used in the benchmarks breaks down as follows: a g5.2xlarge costs US$3.06 per hour in us-east-1, while an equivalent CPU-only c6i.2xlarge (8 vCPUs, 16 GB RAM) costs US$0.34 per hour. These prices are sourced from the AWS EC2 pricing page, accessed September 12, 2024. The GPU instance carries a 9× hourly premium. For a team running a single 50-million-vector index rebuild, the GPU path costs approximately US$0.42 in compute (8.2 minutes at US$3.06/hour), while the CPU path costs US$0.55 (97 minutes at US$0.34/hour). At this scale, the GPU option is cheaper in absolute terms because the speed advantage outweighs the higher hourly rate.

### Break-Even Analysis for Intermittent Indexing Workloads

The cost advantage flips at lower vector counts or for teams that index infrequently and can tolerate longer build times. For a 10-million-vector collection, Milvus 2.4 GPU finishes index construction in roughly 1.7 minutes, costing US$0.09. Elasticsearch 8.14.1 on CPU completes the same task in approximately 18 minutes, costing US$0.10. The absolute savings are negligible at this scale, and the decision hinges on whether the 16-minute time saving justifies maintaining a GPU-enabled Milvus deployment versus a simpler CPU-only Elasticsearch cluster that may already exist in the stack. For teams that rebuild indexes multiple times per day—common in RAG applications with hourly document ingestion from news feeds, support ticket systems, or e-commerce catalogs—the cumulative time savings compound. Four daily rebuilds on a 50-million-vector collection save roughly 5.9 hours of wall-clock time per day, freeing the indexing node for query serving or reducing the required instance count.

### Reserved Instance and Spot Pricing Considerations

The analysis above uses on-demand pricing, but production deployments often use reserved or spot instances. A 1-year all-upfront reserved g5.2xlarge costs approximately US$1.84 per hour, reducing the 50-million-vector index build cost to US$0.25. Spot pricing for g5.2xlarge instances in us-east-1 averaged US$0.92 per hour during August 2024, per AWS Spot Instance Advisor data, which would bring the same build to US$0.13. The CPU instance sees proportional reductions, but the absolute gap narrows. Teams comfortable with spot instance interruption risk—acceptable for batch indexing jobs that can be checkpointed and resumed—can achieve GPU-accelerated index builds at a cost below CPU-only on-demand rates. Milvus 2.4’s index builder supports checkpointing via its segment-based architecture, though the benchmark runs did not test interruption recovery time.

## Algorithmic Trade-offs: IVF_FLAT vs HNSW Under the Hood

The benchmark comparison is not purely a hardware story; the index algorithm choice between Milvus 2.4’s IVF_FLAT and Elasticsearch 8.14.1’s HNSW introduces structural differences that affect both build time and query performance. Understanding these differences is necessary to avoid over-generalizing the benchmark results.

### Why IVF_FLAT Parallelizes Better on GPU

IVF_FLAT partitions the vector space into clusters using k-means, then builds a flat (exact) index within each cluster. The k-means training phase is dominated by matrix multiplication operations that map efficiently onto GPU SIMD units. Milvus 2.4’s GPU index builder, introduced in the June 2024 release, reimplements the k-means assignment and update steps as CUDA kernels that process multiple clusters concurrently. The flat index construction within each cluster is embarrassingly parallel and similarly benefits from GPU acceleration. HNSW, by contrast, constructs a graph layer by layer, with each node insertion requiring a beam search over the existing graph structure. This graph traversal is inherently sequential and branch-heavy, resisting GPU parallelization. Elasticsearch’s HNSW implementation, based on the Lucene 9.10 codec, runs entirely on CPU with no GPU offload path as of version 8.14.1.

### Query-Time Accuracy and Recall Implications

IVF_FLAT with nprobe=32 on the 50-million-vector Deep-1B subset achieved recall@10 of 0.967 in the benchmark, measured against exact brute-force nearest neighbors. HNSW with ef_search=128 achieved recall@10 of 0.981 on the same dataset. The 1.4 percentage point recall gap is statistically significant on this benchmark and reflects the approximate nature of the IVF partitioning; vectors near cluster boundaries can be missed if nprobe is set too low. Teams with strict recall requirements above 0.98 may need to increase nprobe or consider Milvus 2.4’s IVF_PQ index, which was not included in this benchmark. The GPU builder supports IVF_PQ and IVF_SQ8 in addition to IVF_FLAT, with build times scaling proportionally to the compression ratio. A separate benchmark from Zilliz dated July 22, 2024, showed IVF_PQ index build on 50 million vectors completing in 11.4 minutes on the same g5.2xlarge hardware, with recall@10 of 0.951 at 4× compression.

## Operational Considerations Beyond the Benchmark

Synthetic benchmarks on static datasets answer the “how fast” question but leave gaps around production behaviors that affect total cost of ownership. Teams evaluating Milvus 2.4 GPU indexing should weigh several factors that the August 2024 benchmark does not capture.

### Incremental Index Updates and Background Merge Impact

The benchmark measured a from-scratch index build, but production systems rarely rebuild indexes from zero. Milvus 2.4 supports incremental data insertion with a background segment merge process that consolidates small segments into larger ones and rebuilds their indexes. The GPU index builder can be configured to accelerate these background merges, but the benchmark did not test this scenario. Elasticsearch’s segment merge behavior for HNSW vectors is similarly incremental, with Lucene merging segments and rebuilding HNSW graphs on the merged segment. The merge time for both systems depends on segment size and write throughput, and no head-to-head merge benchmark exists as of September 2024. Teams should budget for merge-related index build time in their latency SLO calculations, particularly if the write rate exceeds a few thousand vectors per second.

### Memory Pressure and Multi-Tenancy

Milvus 2.4’s GPU index builder requires the entire vector dataset to fit in GPU VRAM during the clustering phase. For the 50-million-vector dataset, peak VRAM usage was 18.4 GB, which fits on an A10G (24 GB) but would overflow a T4 (16 GB). Larger datasets require either GPU instances with more VRAM (e.g., an A100 with 40 GB or 80 GB) or Milvus’s disk-based index build mode, which falls back to CPU and incurs the performance penalty the GPU path is designed to avoid. Elasticsearch’s HNSW index build uses system RAM and Lucene’s off-heap memory management, which can leverage larger instance memory configurations without a discrete GPU. For a multi-tenant deployment where multiple collections are indexed concurrently, the GPU’s VRAM must be partitioned across collections, and the benchmark’s single-collection focus does not provide guidance on throughput under concurrent index builds.

### Kubernetes and Cloud-Managed Deployment Paths

Milvus 2.4 runs on Kubernetes via the Milvus Operator, and GPU node scheduling requires a Kubernetes cluster with NVIDIA GPU operator installed and GPU node pools configured. Elasticsearch on ECK (Elastic Cloud on Kubernetes) does not require GPU nodes, simplifying cluster topology at the cost of index build speed. Zilliz Cloud, the managed Milvus service, offers GPU-accelerated index builds on its performance-tier instances as of August 2024, with pricing starting at US$0.99 per hour for a GPU-enabled index node. Elastic Cloud does not offer GPU-accelerated vector indexing as of September 2024. Teams that prefer fully managed services and require sub-10-minute index rebuilds on 50-million-vector collections currently have a single-vendor path through Zilliz Cloud; the Elastic Cloud vector search offering is CPU-only.

## Actionable Takeaways

First, measure your actual index rebuild frequency and vector count before defaulting to CPU-only vector databases. If your application rebuilds indexes more than twice daily on collections exceeding 10 million vectors, the Milvus 2.4 GPU path is likely cheaper in absolute cloud compute cost, even at on-demand GPU pricing. Run a 1-week instrumentation pass on your current system to capture the real rebuild cadence and dataset growth rate.

Second, do not assume GPU instances are always more expensive. The break-even analysis above shows that at 50 million vectors, the GPU index build costs US$0.42 versus US$0.55 for CPU, because the 11.8× speed advantage outweighs the 9× hourly rate premium. Spot instances widen this gap further. Calculate the cost for your specific vector count using the formula: (your_vector_count / 50,000,000) × 8.2 minutes for GPU, and (your_vector_count / 50,000,000) × 97 minutes for CPU, multiplied by your instance pricing.

Third, if your recall requirements exceed 0.98, benchmark IVF_FLAT with nprobe values between 64 and 128 on your own dataset before committing to the GPU index path. The 1.4 percentage point recall gap observed in the August 2024 benchmark may be unacceptable for applications like legal document retrieval or medical literature search where missed results carry regulatory risk. The recall gap can be closed by increasing nprobe at the cost of higher query latency, and you should measure that trade-off on your own data distribution.

Fourth, plan your GPU VRAM budget around peak dataset size, not average. The 50-million-vector benchmark used 18.4 GB of an A10G’s 24 GB. If your collection is projected to reach 65 million vectors within the GPU instance’s lifecycle, an A10G will overflow, and you will need either an A100-40GB instance or a migration to disk-based indexing. Factor VRAM headroom into your instance selection at provisioning time, not when the out-of-memory error hits during a production index rebuild.

Fifth, if your stack already includes Elasticsearch for text search and observability, and your vector search workload is under 10 million vectors with infrequent rebuilds, the operational simplicity of keeping vectors in Elasticsearch outweighs the GPU speed advantage. The cost difference is negligible at that scale, and maintaining a separate Milvus deployment adds cluster management overhead that is not justified by an 18-minute versus 1.7-minute index build time when builds run once daily. Switch to Milvus GPU indexing when rebuild time starts blocking your deployment pipeline or when query latency under concurrent index builds exceeds your p99 SLO.
