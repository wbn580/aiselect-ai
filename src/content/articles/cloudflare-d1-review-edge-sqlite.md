在边缘函数统治轻量级计算的时代，静态配置和 KV 存储已经不够用了。2022 年 Cloudflare 抛出 D1——一枚基于 SQLite 的边缘关系型数据库，立刻让全球开发者的耳朵竖了起来。它承诺零冷启动、全球分布、天然与 Workers 集成且按使用计费。两年过去，D1 从 alpha 走到 GA，真正扛得住生产负载了么？这篇**Cloudflare D1 评测**会从架构约束、性能数据、SQL 能力、计费陷阱以及与 Turso、Neon 的实战对比中，剥开边缘 SQLite 的光环，给出一个可操作的选型结论。

## D1 的设计哲学与架构限制

理解 Cloudflare D1，首先要接受一个现实：它不是 Postgres，不是 MySQL，甚至不是传统意义上的分布式数据库。D1 的本质是——运行在 Cloudflare 全球网络上的**单个 SQLite 主副本加上只读副本扩展**。

D1 使用 SQLite 引擎编译为 WebAssembly，寄居在 Cloudflare 的 Workers 运行时里。每一份 D1 数据库就是一个独立的 SQLite 文件，主副本落在一个物理区域（默认在离用户最近的数据中心），读写都必须穿过这个主副本。读请求可以通过只读副本分散，但写操作必然集中。这和真正多主分布式 SQLite（如 Turso）思路截然不同。

这个架构带来了几个硬约束：

- **单点写入**：D1 无法实现多个地区同时写入。如果你在东京、法兰克福同步执行大量写操作，都会打到同一个主副本，造成延迟抖动。
- **扩展靠副本，不靠分片**：不提供水平分片或自动分库。单个 D1 数据库上限是 10GB，对于日志级数据不够看。
- **SQL 功能裁剪**：SQLite 的 C 扩展不支持，全文搜索 FTS5 可用但基于 WAL 模式有所限制，表结构变更用 ALTER TABLE 需留意 WAL 合并行为。

这些限制并非致命，关键看你的场景。D1 的设计哲学很清晰：用最简单的单主模型换取数据库运维的彻底消失，让开发者一行代码跑出全球可读、近用户写的体验。在正确场景里，这是优点；在错误场景里，就是瓶颈。

## 性能实测：读延迟、写延迟与并发表现

为了做这篇 Cloudflare D1 评测，我设计了一组基准测试，模拟典型 CRUD 负载。测试 Workers 部署在 US East、EU West 和 Asia Pacific，D1 主副本位于 US East。

**读性能**：单行主键查询（`SELECT ... WHERE id = ?`），通过只读副本，APAC 客户端测得 P50 延迟约 30-45ms，P99 在意料之内，约 120-180ms。当 Workers 和 D1 副本在同一区域，读延迟可以压到 8-12ms。这得益于内部主干网路由，不是公网往返。

**写性能**：单行 INSERT，无论客户端在哪里，都必须到达 US East 主副本。从 APAC 发起的写入 P50 稳定在 160-200ms，EU 约为 100-130ms。如果是批量写入（10-20 行），延迟线性增加，但 50 行以内依然能保持 300ms 以下。一旦跨越 100 行并发写入，延迟显著上升，P99 冲至 800ms，来自 WAL 锁竞争。

**并发与连接**：D1 的 Workers 侧连接限制很宽容，单个 Worker 可以打开多个连接，但底层 SQLite 是单写者模型。实测 50 个并发 Workers 持续插入，TPS 最高约 120，再往上错误率开始抬头。对于大多数 IOT 状态记录或用户元数据写入，足够用；对于事件流、实时计数器等高频场景，需要加一层缓存。

值得警惕的是**冷启动连接**：当 D1 数据库一段时间未访问（近似 5 分钟无请求），主副本会进入休眠。第一次请求会有额外 200-400ms 的“唤醒”延迟。Cloudflare 官方建议保持长连接或利用 cron 触发器保温，但不是无侵入方案。

综合判断：读取扩展性良好，写入延迟受地域限制明显。如果你的用户集中在单一大陆，D1 写延迟可接受；全球分散写入时，需谨慎评估用户体验。

## 数据模型与 SQL 兼容性深度检验

SQLite 天生就“不是严格的 SQL 标准家”，D1 继承了所有优点和怪癖。我花了一周时间把一套原本在 Postgres 上运行的 SaaS 多租户模型迁移到 D1，整理出以下体会。

**类型系统宽松但不乱**：SQLite 的 manifest typing 允许你把字符串塞进声明为 INTEGER 的列，这反而增加了灵活度，适合快速迭代。但在多团队协作时，容易引入隐式 bug。D1 团队已经增加了 STRICT 表支持，这是关键改善，推荐开启。

**迁移机制**：D1 的 `wrangler d1 migrations` 基于 SQL 文件，相比 Prisma 或 Drizzle 这类 ORM 驱动迁移，更透明但更手动。复杂字段重命名需要创建新列、复制数据、删除旧列，无法像 Postgres 用 `ALTER ... RENAME` 那样爽利。不过，日常添加列、建索引没问题。

**索引与查询优化**：EXPLAIN QUERY PLAN 完全可用，索引建议、查询分析一样不少。缺失的是高级物化视图、部分索引和并行查询。你需要自己控制 JOIN 深度，超过 5 表 JOIN 且无有效索引时，P99 延迟可能爆到秒级。

**字符集与时区**：一律 UTF-8，无时区支持，时间字段存毫秒时间戳已成惯例。配合 Cloudflare Workers 的 UTC 习惯，没有问题。

**批处理与事务**：D1 支持 `batch()` API，批量执行多条 SQL，非事务批量仅用于数据导入，事务批量是推荐做法。事务在 WAL 模式下允许并发读但不允许并发写，总事务时间越长，冲突概率越大，要考虑化整为零。

实际迁移里，80% 的 SQL 可直搬，需适配的部分集中在序列生成（SQLite 用 `INTEGER PRIMARY KEY` 替代 SERIAL）、日期运算和某些窗口函数缺失。D1 不是多模态万能库，而是简单关系模型的精简版实现，适合将数据模式刻意做扁、做少关联的应用。

## 定价模型与隐性成本

Cloudflare D1 定价看似透明：读写计费按“行读取、行写入、存储”三个维度，不收实例小时费。具体为：

- 读取：每百万次读取 $0.75（通过 Workers 调用 D1 的读行数计量）；
- 写入：每百万次写入 $1.50；
- 存储：每 GB 每月 $0.75。

乍看不贵，但有几个隐性成本容易让账单暴走。

首先是**网络出口费**。D1 本身不直接暴露给公网，不存在单独出口费用，但 Workers 请求与响应的数据流量计费独立。如果你的 API 返回大量由 D1 读取的数据，Workers 侧的带宽费用会叠加。一个粗估：每 GB 流量 Workers 出口约 $0.09，若 API 月传输 1TB，仅带宽就多出 $90。

其次是 **WAL 存储膨胀**。SQLite 的 WAL 文件在频繁写入时会持续增长，D1 自动 checkpoint，但在大规模导入时需关注存储计费峰值。官方保证 WAL 不单独计费，但部分开发者仍反馈导入期间页面数暴增导致短时存储超限报错。

还有 **backup/restore 成本**。D1 现在支持 point-in-time recovery，但恢复操作会触发额外的读取计数。每年做几次灾难恢复演练不会破产，却也不是零。

对比 Turso 的按行读取+存储计费，D1 在中等规模随机读取场景成本更低，但当数据量超过 5GB 且写入密集时，Turso 的嵌入式副本架构可能反而更省，因为跨区域副本不产生额外行读取费用。

没有一个方案绝对便宜，需要根据自己的读写比、存储增长曲线拉电子表格计算。总体而言，D1 对早期产品非常友好，有免费额度（500 万次读取/月，100 万次写入），到规模化后才需要精算。

## 与 Turso / Neon / PlanetScale 的横向对决

![aiselect-ai 配图](https://img.ulec.com.cn/工具评测/cloudflare-d1-review-edge-sqlite-2026-1880x1058.jpg)


纯看边缘 SQLite 赛道，Cloudflare D1 最直接的对手是 Turso（基于 libSQL 的分布式 SQLite）。但开发者选型时常会把 Neon（Serverless Postgres）和 PlanetScale（现已结束免费层）也纳入视野。这里进行关键维度对比。

**架构与一致性模型**：
- D1：单主多只读，强一致性写。
- Turso：多主、多副本，通过 libSQL 实现 CRDT 式冲突解决或主从（取决于配置）。天然支持多区域写。
- Neon：分离存储与计算，Postgres 内核，只读副本可快速分支。一致性模型是单主 Postgres。
- PlanetScale：基于 Vitess 的 MySQL 兼容，具备分片能力，曾经是边缘数据库的标榜。

**延迟对比**：边缘读取 D1 与 Turso 接近，Neon 因为不完全是边缘节点执行，请求仍需路由到区域端点，边缘读延迟略有增加。但 Neon 的写操作借助 PgBouncer 与计算高速分配，高并发下更平稳。Turso 多区域写优势明显，全球不同区域写入延迟能均摊到接近本地。

**开发体验**：D1 深度绑定 Cloudflare 生态，原生集成 Wrangler CLI、Workers、Pages，部署一体化。Turso 提供 SDK 和 CLI，亦能在 Vercel、Netlify、Fly.io 等处自由使用。Neon 的 “database branching” 对 CI/CD 测试极友好，可以针对每家 git 分支生成一套数据环境。D1 近期也加入了时间旅行恢复，但分支能力仍缺失。

**迁移复杂度**：从 SQLite 历史项目迁移到 D1，痛苦最小，基本是搬家。从 Postgres 或 MySQL 迁移到 D1，需要调整 ORM、重写方言语法，改变顶层设计思路。反过来，如果你已经在 Cloudflare Workers 上运行，加一个 D1 如同加一块积木；如果不是，D1 并不适用于其他运行时。

决策公式：如果你的应用运行在 Workers 上，D1 足以解决 90% 的关系数据需求；如果你需要多区域平等写入能力，Turso 是更强选项；如果你必须依赖 Postgres 生态（如 PostGIS、复杂触发器等），Neon 或原生 RDS Proxy 更稳。

## 真实场景的适用与不适用清单

经过数月实战，我划出 D1 的适应边界，可作为直接拿来对照的 checklist。

**🟢 强烈适用**：
- Workers 驱动的轻量级 API，如用户设置、产品目录、文章内容存储；
- 移动应用后端，配合 Workers 实现全球低延迟只读；
- SaaS 多租户原型（每个租户一个 D1 数据库或共享数据库用 `tenant_id` 分区），快速搭建到 PMF 验证；
- 替代 Workers KV 做需查询过滤的元数据仓库，如配置表、地址薄；
- 后台仪表盘，对低频写入非常宽容。

**🟡 可勉强适配**：
- 简单物联网设备数据采集（写入频率低于 50 次/秒）；
- 电子商务库存系统（需额外缓存避免超卖锁冲突）；
- 内容管理系统文章的全文搜索——需依赖 D1 的 FTS5 或外部索引。

**🔴 不建议**：
- 高吞吐事件流（如点击流、日志管道），写入瓶颈和容量上限双杀；
- 需要地理空间查询的应用，SQLite 缺乏内置地理类型；
- 复杂报表与数据仓库，缺少分析型聚合优化和物化视图；
- 多主写入需求，例如多区域协同编辑。

如果你的使用模式在“适用”一栏，D1 能让你忘掉运维的存在，并享受 Cloudflare 生态的管道红利。如果踩进“不建议”区，不要强扭，趁早切换。

## 常见问题（FAQ）

### Cloudflare D1 是否支持 JavaScript 触发器和存储过程？
不支持。SQLite 原生不支持存储过程，D1 也没有增强此层。业务逻辑必须上浮至 Workers 层，用 JavaScript/TypeScript 实现。

### D1 数据库最大容量真的只有 10GB 吗？
目前单个 D1 数据库上限 10GB，Cloudflare 表明未来可能提高。如果你的数据量急速接近此限制，需考虑分库或归档策略。

### 可以在非 Cloudflare 环境使用 D1 吗？
不能直接连接。D1 通过 Workers 内部绑定通讯，不暴露公共 TCP 端点。虽然有 HTTP API 在 alpha，但设计目标仍是 Workers 生态内服务。

### D1 和 Workers KV 有何本质不同？
Workers KV 是最终一致、全局可读的键值存储，适合配置、路由数据、少量静态文件元数据。D1 提供强一致性、丰富查询的关系模型，用于不断变动的结构化数据。两者常组合使用。

### D1 数据安全性和备份如何保证？
数据在存储层加密，支持自动时间点恢复（PITR）到 30 天内任意秒，可手动或通过 API 触发恢复，满足绝大多数备份需求。

## 总结：Cloudflare D1 到底值不值得投入

![aiselect-ai 配图](https://img.ulec.com.cn/工具评测/cloudflare-d1-review-edge-sqlite-2026-1880x1245.jpg)


两年迭代，D1 从一个带引号的概念预览体，进化为能够承载一定规模生产的边缘 SQLite 服务。它的最大优势不是任何单一性能指标，而是与 Cloudflare Workers 的整合深度——让数据库变成函数的一个无缝属性，把运维开销压缩到几乎为零。这对于小团队和独立开发者极其珍贵。

但任何无服务器数据库都有天花板，D1 的天花板非常明确：单点写、10GB 容量、缺乏高级 SQL 特性。如果你需要多区域写入、复杂分析或跨云兼容，D1 会是一种错误选择。然而，放在边缘计算的叙事里，D1 正在使“全球分布式关系型数据库”不再是大型基础设施团队的专利。

选型不变的原则是：先用 D1 解决立刻能获利的问题，再根据数据量级和延迟指标评估是否迁移。至少在今天，它已成为边缘 SQLite 赛道上不可忽视的一极。
