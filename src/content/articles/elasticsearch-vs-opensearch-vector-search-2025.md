In mid-2025, the vector search conversation has shifted. The question is no longer whether a general-purpose search engine can handle embeddings at scale — both Elasticsearch and OpenSearch proved that years ago. The question is how each platform handles the operational reality of hybrid ranking, where sparse BM25 scores and dense vector similarities must be combined into a single result set without silently degrading recall. This matters because the cost of getting hybrid search wrong is not theoretical. An e-commerce site that mixes keyword relevance with semantic similarity incorrectly will surface products that look right to a model but miss the user’s explicit intent. A legal document retrieval system that weights vectors too heavily will return conceptually related but jurisdictionally irrelevant case law. The licensing divergence between the two projects, which began in 2021 when Elastic changed the license on Elasticsearch 7.11 and newer releases, continues to shape procurement decisions inside enterprises where legal review of open-source dependencies is now standard operating procedure. On the performance side, the gap in raw vector throughput between the two engines has narrowed considerably since 2023, but the differences in how each implements quantization, segment-level parallelism, and filter pushdown during ANN queries remain substantial enough to influence architecture choices for teams building retrieval-augmented generation pipelines with latency budgets under 200 milliseconds.

## Licensing and Distribution: The Fork’s Practical Impact in 2025

The licensing split that created OpenSearch as an AWS-led fork of Elasticsearch 7.10.2 is now over four years old, and the downstream effects are no longer speculative. Teams evaluating the two engines in 2025 are making decisions based on accumulated divergence in features, not just philosophical alignment with the Apache 2.0 license.

### What the License Change Actually Blocks

Elasticsearch versions from 7.11 onward ship under the Elastic License 2.0, which is not an open-source license as defined by the Open Source Initiative. The practical restriction that trips procurement teams is Section 1, Limitation (b): the software may not be provided as a managed service to third parties. For any company building a SaaS product that exposes search or vector capabilities to its own customers, running a managed Elasticsearch instance internally is permitted, but wrapping Elasticsearch and reselling it as a hosted search service is explicitly prohibited. OpenSearch remains under Apache 2.0 with no such restriction. This distinction alone has pushed a measurable number of B2B SaaS startups toward OpenSearch, not because of cost — Elastic Cloud pricing is competitive at scale — but because the legal ambiguity of what constitutes a “managed service” creates friction during fundraising due diligence and enterprise contract reviews.

### Feature Divergence Since the Fork

As of June 2025, the two projects have diverged in ways that go beyond license text. Elasticsearch has invested heavily in its proprietary Learned Sparse Encoder for hybrid ranking, introduced in version 8.12 (January 2024), which generates sparse vectors from a trained model rather than relying solely on BM25 tokenization. OpenSearch has no equivalent; its hybrid search relies on combining BM25 scores with dense vector similarity using arithmetic or harmonic mean fusion. On the OpenSearch side, the neural search plugin introduced in version 2.9 (July 2023) and matured through 2.16 (September 2024) provides native integration with external embedding models via the ML Commons plugin, including support for Cohere Embed v3 and Amazon Titan Text Embeddings v2. Elasticsearch’s inference API, redesigned in 8.11 (November 2023), supports a broader range of embedding providers natively, including OpenAI text-embedding-3-large and voyage-3, without requiring external middleware.

### Enterprise Support and Vendor Lock-In

Elastic N.V. provides first-party support and managed services for Elasticsearch through Elastic Cloud, available on AWS, GCP, and Azure. OpenSearch has no single commercial entity behind it; AWS offers Amazon OpenSearch Service as a managed solution, and third-party providers like Aiven and Instaclustr offer alternative managed OpenSearch deployments. The multi-vendor support story for OpenSearch is stronger in 2025 than it was in 2022, but the absence of a single project steward means that feature roadmaps are influenced by AWS priorities, which skew toward integration with the AWS ecosystem — SageMaker for model hosting, Bedrock for embedding generation, IAM for authentication. Teams that want a search engine that is cloud-agnostic in both deployment and feature availability increasingly default to Elasticsearch, accepting the license terms as a known quantity rather than navigating the AWS-centric gravity of OpenSearch development.

## Vector Indexing Performance: HNSW Configuration and Quantization

Both engines implement the Hierarchical Navigable Small World (HNSW) algorithm for approximate nearest neighbor search, but the configuration defaults and quantization options differ enough to produce double-digit percentage gaps in indexing throughput and query latency under identical hardware profiles.

### HNSW Defaults and Their Consequences

Elasticsearch 8.16, released in December 2024, sets default HNSW parameters of `m=16` and `ef_construction=100` for the `dense_vector` field type with `index: true`. OpenSearch 2.16 defaults to `m=16` and `ef_construction=128`. The higher `ef_construction` value in OpenSearch produces a higher-quality graph at the cost of longer indexing times. In benchmarks published by Elastic on November 12, 2024, using the DBPedia entity dataset with 1,000-dimensional vectors and a recall target of 0.99, Elasticsearch 8.15 indexed 1 million vectors in 47.3 minutes on a 3-node cluster with 64 GB RAM per node, while OpenSearch 2.14 required 62.1 minutes on identical hardware. When `ef_construction` was normalized to 100 on both engines, the indexing time gap narrowed to 47.3 versus 51.8 minutes, suggesting that approximately 70% of the observed difference was attributable to the default parameter difference rather than fundamental engine architecture. The remaining 30% gap was attributed to Elasticsearch’s segment-level parallelism during bulk indexing, which OpenSearch’s segment merging strategy does not replicate identically.

### Scalar Quantization and Memory Footprint

Scalar quantization, which maps float32 vector components to int8 values, is available in both engines but with different implementation maturity. Elasticsearch introduced native scalar quantization for `dense_vector` fields in version 8.13 (March 2024) with the `index_options.type: int8` setting. OpenSearch followed with byte vector support in version 2.14 (June 2024) via the `faiss` engine’s `encoder.flat` configuration. In practice, enabling int8 quantization reduces per-vector memory from 4,096 bytes (for a 1,024-dimensional float32 vector) to 1,024 bytes, plus HNSW graph overhead. For a corpus of 100 million vectors, this translates to approximately 306 GB of RAM for float32 versus 76.5 GB for int8, excluding graph edges. Elasticsearch’s implementation stores the quantized vectors on disk and loads them into the page cache on demand, while OpenSearch’s FAISS-based approach keeps the quantized vectors in process memory. The Elasticsearch approach permits larger-than-memory vector indices at the cost of page fault latency on cold queries; the OpenSearch approach guarantees consistent query latency but imposes a hard ceiling on vector count per node based on available RAM. A July 2024 benchmark by the OpenSearch project, published on opensearch.org/blog, showed that int8 quantization on OpenSearch 2.15 achieved 98.5% recall@100 compared to float32 at 99.2% on the ANN-Benchmarks SIFT-1M dataset, with query latency dropping from 12.4 ms to 8.7 ms at the 99th percentile.

### Disk-Backed Vector Search

Elasticsearch 8.15 (August 2024) introduced disk-backed vector search as a generally available feature, allowing `dense_vector` fields to be indexed with `index_options.type: int8_hnsw` where the HNSW graph resides on disk rather than in the JVM heap or page cache. This is distinct from memory-mapped disk access; the graph structure itself is stored in a format optimized for sequential reads from NVMe storage. In testing with 500 million 768-dimensional vectors on a 10-node cluster with 7.68 TB NVMe drives per node, Elasticsearch reported median query latency of 94 ms at 95% recall@10, compared to 22 ms for an in-memory int8 index of the same size on a cluster with sufficient RAM. The trade-off is explicit: disk-backed search enables vector indices that exceed aggregate cluster memory by 5-10x at the cost of 4-5x higher query latency. OpenSearch has no equivalent feature as of version 2.16; vectors must fit in process memory or be offloaded to an external FAISS index managed outside the OpenSearch process, which breaks the atomicity of document updates and hybrid queries.

## Hybrid Search and Reciprocal Rank Fusion

Hybrid search — combining lexical (BM25) and semantic (dense vector) relevance signals — is the default retrieval strategy for most RAG pipelines in 2025. Both engines support it, but the fusion algorithms and score normalization approaches differ in ways that affect result quality for queries where lexical and semantic signals disagree.

### Elasticsearch’s RRF Implementation

Elasticsearch introduced Reciprocal Rank Fusion (RRF) as a built-in retriever in version 8.9 (July 2023) and matured it through 8.16. RRF combines result sets from multiple queries by scoring each document as `1 / (k + rank)`, where `k` is a constant (default 60) that controls the influence of low-ranked documents. A document ranked 1st by BM25 and 5th by vector search receives a combined score of `1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318`. The key property of RRF is that it does not require score normalization between the two retrieval methods; BM25 scores are unbounded and vector similarity scores depend on the embedding model’s output range, but ranks are always integers, making RRF robust to score distribution differences. Elasticsearch 8.16 extends RRF with the `rank_constant` parameter adjustable per query, allowing tuning of the fusion behavior without reindexing. A `rank_constant` of 10 gives more weight to high-ranked documents; a value of 100 produces flatter score distributions that surface more diverse results.

### OpenSearch’s Normalization-Based Fusion

OpenSearch implements hybrid search through the `hybrid` query type introduced in version 2.11 (October 2023), which supports `min_max`, `l2`, and `zscore` normalization methods before combining scores with a weighted arithmetic mean. The `min_max` method scales scores to [0,1] using the observed minimum and maximum scores in each sub-query result set. The `l2` method normalizes scores to unit vectors and computes Euclidean distance. The `zscore` method standardizes scores to zero mean and unit variance. Each normalization method makes assumptions about score distributions that can break under certain query patterns. `min_max` is sensitive to outliers; a single BM25 score of 450 when all others are below 50 will compress the normalized scores of all other documents into a narrow band near zero, effectively eliminating BM25’s contribution to the final ranking. `zscore` assumes a roughly normal distribution of scores, which holds for BM25 on large text corpora but frequently fails for vector similarity scores from models like text-embedding-3-large, which produce cosine similarities tightly clustered between 0.65 and 0.95 for relevant documents.

### Benchmarking Fusion Quality

A comparative evaluation published by Mixedbread on January 15, 2025, tested both engines on the BEIR benchmark’s NFCorpus dataset using the same embedding model (mixedbread-ai/mxbai-embed-large-v1) and identical BM25 configurations. Elasticsearch 8.15 with RRF (k=60) achieved NDCG@10 of 0.3521. OpenSearch 2.15 with `min_max` normalization and equal weighting achieved NDCG@10 of 0.3418. The 0.0103 NDCG difference is statistically significant (p < 0.01, paired t-test across 323 queries) but practically small — roughly equivalent to swapping one relevant document in the top 10 for a marginally relevant one across 3% of queries. When OpenSearch used `zscore` normalization instead of `min_max`, NDCG@10 rose to 0.3487, narrowing the gap to 0.0034 and falling within the margin of noise. The practical conclusion is that fusion method matters less than consistent parameterization; both engines can achieve comparable hybrid quality when properly configured, but Elasticsearch’s RRF requires fewer decisions about normalization strategy and is less sensitive to score distribution anomalies.

## Cost and Operational Complexity in 2025

Procurement decisions in mid-2025 are shaped as much by operational overhead as by raw benchmark numbers. The fully loaded cost of running either engine in production includes compute, storage, network egress, and the engineering time required to maintain the cluster.

### Managed Service Pricing

Elastic Cloud pricing as of June 2025 starts at $95 per month for a single-node deployment with 1 GB RAM and 16 GB storage, which is suitable only for development. A production-grade 3-node cluster with 64 GB RAM and 512 GB storage per node in the AWS us-east-1 region costs $2,847 per month under the Standard tier, which includes 24/7 support with a 4-hour response SLA. The Enterprise tier, which adds 1-hour response SLA and dedicated support engineering, adds 20% to the base cost, bringing the monthly total to $3,416.40. Amazon OpenSearch Service pricing for an equivalent 3-node `r6g.4xlarge.search` cluster with 128 GB RAM and 512 GB EBS gp3 storage per node runs $2,304 per month in us-east-1, excluding data transfer costs. The $543 monthly difference per cluster, or $6,516 annually, is meaningful for teams running multiple clusters across environments but is unlikely to be the deciding factor for organizations where a single production cluster serves the primary search workload.

### Self-Managed Cost Structure

Self-managed deployments shift the cost from service fees to engineering time. A 3-node cluster on reserved EC2 instances (r7i.4xlarge, 3-year all-upfront) costs approximately $1,087 per node per month in us-east-1, or $3,261 per month for the cluster, plus EBS storage at $0.08 per GB-month for gp3 volumes. At 512 GB per node, storage adds $122.88 per month, bringing the infrastructure subtotal to $3,383.88. This is within 5% of the managed Elastic Cloud Standard tier pricing, meaning the decision to self-manage is primarily about control — access to JVM settings, custom plugins, and deployment topology — rather than cost savings. The engineering overhead of managing a self-hosted cluster, estimated at 0.25 to 0.5 full-time equivalent for a production deployment with HA requirements, adds $37,500 to $75,000 annually at a fully loaded engineer cost of $150,000, which dwarfs the infrastructure savings from avoiding managed service margins.

### Hidden Costs: Embedding Generation and Refresh Overhead

A cost factor that does not appear in cluster pricing pages is embedding generation. Both engines can call external embedding services at query and index time, but the architectural choice of where embeddings are generated affects latency and cost. Elasticsearch’s inference processor, introduced in 8.11, generates embeddings within the ingest pipeline, adding approximately 80-120 ms per document when calling OpenAI text-embedding-3-large with 1,024 dimensions at $0.00013 per 1,000 tokens. For a corpus of 10 million documents averaging 512 tokens each, the embedding generation cost is approximately $665.60, plus the compute cost of the ingest nodes. OpenSearch’s ML Commons plugin can host embedding models directly on the cluster, eliminating external API calls at the cost of GPU-equipped data nodes. An Amazon OpenSearch Service cluster with `g5.2xlarge` GPU nodes adds $1.52 per node-hour, or $1,094.40 per month per GPU node, which must be weighed against the external API costs and the latency penalty of cross-service calls.

## Actionable Recommendations

First, choose OpenSearch if your application exposes search functionality as a managed service to third parties. The Apache 2.0 license eliminates the legal ambiguity that the Elastic License 2.0 creates for SaaS resellers, and the feature gap between the two engines in vector search is small enough that license compliance should dominate the decision. Second, choose Elasticsearch if your team values operational simplicity in hybrid search configuration. RRF eliminates the normalization strategy decisions that OpenSearch requires, and the risk of misconfiguring score normalization and silently degrading result quality is real for teams without dedicated search engineers. Third, enable scalar quantization from day one if your vector corpus exceeds 10 million documents. The recall loss of 0.5-1.5 percentage points is offset by a 75% reduction in memory footprint, and both engines now support int8 quantization as a stable feature. Fourth, budget for embedding generation costs separately from cluster infrastructure. Whether you pay an external API per token or provision GPU nodes for local inference, embedding generation at scale is a line item that frequently surprises teams migrating from keyword-only search to hybrid retrieval. Fifth, test hybrid fusion quality on your own queries before committing to either engine. The NDCG differences observed in published benchmarks are small enough that domain-specific query patterns — short queries, long queries, queries with numeric filters, queries with negations — will dominate the real-world quality difference between the two platforms.
